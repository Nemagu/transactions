# Инструкция для работы с сервисом

## Архитектура
Сервис разделен по принципам гексогональной архитектуры.
Для проектирования domain слоя используется подход ddd, для application слоя используется CQRS.
Domain слой должен быть полностью изолирован от слоев ниже по иерархии, для связи можно использовать только интерфейсы. Аналогичный подход должен использоваться и для application слоя.

## Зависимости
Для презентационного слоя используется FastAPI (валидация данных через pydantic), nats. В адаптерах используется psycopg, nats-py. Инструмент для миграций реляционной базы данных - yoyo. Для тестирования используется pytest с плагинами pytest-asyncio, pytest-cov. Форматировщик кода - ruff.
В production будет использоваться база данных postgres 18 версии, брокер сообщений - nats с jetstream.
Для получения переменных окружения используется pydantic_settings.
Все зависимости находятся в виртуальном окружении в корневой дирекотрии в .venv/. Так же работа с зависимостями возможна через uv.

## Документирование кода
- Каждый класс, метод, функция должны быть задокументированы подробно. Они должны иметь описание, секцию с описанием аргументов, исключений, который может вызвать текущая структура и возвращаемого типа по правилам python.
- Документация должна быть написана на русском языке.
- Для docstring должен использоваться единый стиль по всему репозиторию:
  краткое описание, затем при необходимости секции `Args:`, `Raises:`, `Returns:`.
- `property` и `classmethod` так же должны быть задокументированы.
- Для `__init__` краткое описание не требуется. В docstring инициализатора следует сразу начинать с секций, например с `Args:`.
- Если для структуры не применимы `Args:`, `Raises:` или `Returns:`, то нужно указывать только релевантные секции.
- При документировании нового или изменяемого кода необходимо ориентироваться на уже существующую документацию в текущем репозитории и продолжать тот же стиль без отдельного запроса примера у пользователя.
- Если в файле отсутствует документация, то при его редактировании ее необходимо добавить как минимум для всех измененных публичных сущностей, а для базовых и архитектурно значимых сущностей - полностью.
- Пример оформления:
```python
class ExampleService:
    """Сервис выполнения примера."""

    def __init__(self, value: int) -> None:
        """
        Args:
            value (int): Значение для инициализации сервиса.
        """
        self._value = value

    @property
    def value(self) -> int:
        """
        Returns:
            int: Текущее значение сервиса.
        """
        return self._value

    def execute(self, increment: int) -> int:
        """Выполняет действие сервиса.

        Args:
            increment (int): Величина увеличения значения.

        Raises:
            ValueError: Передано отрицательное значение `increment`.

        Returns:
            int: Обновленное значение.
        """
        if increment < 0:
            raise ValueError("increment не может быть меньше 0")
        self._value += increment
        return self._value
```
- Если перенос строк требует форматер, то перенос должен быть осуществлен через символ "\".

## Правила разбиения решение задачи на части
- Если задача представляет собой объемную работу, то необходимо разбивать ее решение на более простые задачи и решать их шаг за шагом, задачу за задачей по отдельности.
- Размер решения задачи не должен превышать 50-60 строк кода!, если решение превышает этот размер, то задача должна быть разбита на еще более мелкие.
- Каждый следующий шаг не должен быть начат если не согласован текущий.
- Если пользователь не предоставил план действий, то следует сформировать его самостоятельно и согласовать этот план с пользователем.
- Если план план пользователя нарушает какие либо требования, то следует уведомить его.
- План пользователя и его требования имеют большей приоритет над требованиями, которые указаны в этом файле.

## Правила тестирования
Все тесты необходимо располагать в текущем репозитории в ./src/tests. Тесты должны быть разбиты на юнит-тесты и интеграционные тесты. Для тестирования презентационного слоя не требуются юнит-тесты, для остального product кода такие тесты необходимы.

- Для создания агрегатов, команд и т.п. необходимо использовать фабрики, которые должны быть вложены в фикстуры.
- Если создается очередная фабрика в тестовом файле, а не файле conftest.py, то необходимо убедиться что такая фабрика не существует в других тестовых файлах. Если фабрика существует, то следует ее вынести в файл conftest.py в соответствии с другими требованиями, указанными в этом файле.
- Если файла conftest.py не существует на необходимом уровне, его необходимо создать.
- Если файл conftest.py после редактирования остается пустым, то его необходимо удалить.
- Дублирование фикстур не допускается. Если фикстуру необходимо использовать в нескольких модулях, то ее следует вынести на минимально высокий общий уровень вверх по иерархии.
- Схожие тесты необходимо объединять с помощью декоратора parametrize. Каждый такой декоратор должен включать параметр ids, чтобы название тестов корректно отображалось.
- Процент покрытия тестами должен быть не менее 70 процентов, для domain слоя - 100 процентов.
- Тестировать требуется только тот код, который написан в текущем репозитории. Т.е. тестирование фреймворков, зависимостей и т.п. не допускается.

- Для тестирования допускается создание упрощенных реализаций интерфейсов. Например, для работы с базой данных (unit of work). Интеграционные тесты не должны использовать моковые реализации, для них должна подниматься инфраструктура.

- Для запуска инфраструктуры для тестирования необходимо использовать подпроцессы, пользователь не должен вручную запускать создание инфраструктуры.
- Файлы для создания инфраструктуры не должны храниться в репозиториях, они должны создаваться в папке /tmp во время старта тестирования.
- После завершения тестирования вся созданная инфраструктура должна быть остановлена и по необходимости все связанные с ней временные файлы должны быть удалены.
- С применением миграций должен работать только выбранный для этого инструмент. Костыли в виде ручного чтения файла миграций и применение этого текста как sql команды и т.п. не допускается.
- Переменные для работы с настройками приложения должны устанавливаться и быть получены из переменных окружения.

## Правила написания кода
- Все абстрактные методы не должны содержать в своей области видимости какого либо кода. Они должны содержать в себе только "...".
- При создание экземпляра класса исключения все аргументы должны передаваться через имя. Аргументы с значением по умолчанию допускается не передавать, если значение по умолчанию идентично необходимому для передачи.
- Для работы с базой данных запрещено использовать f строки.
- Не используемые импорт должен быть удален из файла.
- Приоритет отдается импорту конкретных функций из модуля, а не модулю целиком. Модуль можно импортировать целиком, если от туда нужно много элементов.

## Архитектурные требования к коду
- Все объекты значения должны быть неизменяемыми.
- Редактировать приватные поля агрегата может только агрегат (начинаются с "_" и "__"). Исключительные случаи возможны только в рамках тестирования при невозможности иного пути.

## Действия после завершения редактирования кода
- Необходимо проверить форматирования кода. Команда "uv run ruff check"
- Необходимо убедиться что все тесты пройдены. Команда "uv run pytest"
